# BOSS v9.2.4

**Release Date:** 2026-06-02

Plugin loading and validator fixes: locally-built plugins newer than the published release now actually load (the bundled-plugins path was scanning the wrong directory, and the background updater silently downgraded ahead-of-release local builds), and `BinaryCompatibilityValidator` no longer disables a plugin over signature drift in its *bundled* third-party libraries — only the `ai.rever.boss.plugin.*` contract is enforced.

## Highlights

- **Local plugin dev builds actually load** (#754): `copyBundledPluginsLocal` was scanning `../../boss_plugin/...` (singular) but the repo is `boss_plugins` (plural), so the bundled copies stayed stale and local builds never entered the pipeline (and the "Source directory not found" warnings recurred). Separately, `scheduleBackgroundUpdateCheck` downloaded the published GitHub release whenever `installedVersion != latestVersion` with no newer-than check, so a local build at e.g. `2.2.0` was downgraded to published `2.1.9` and `downloadSystemPluginFromGitHub` then deleted the local jar. The download is now guarded with `isNewerVersion(latest, installed)`.
- **Validator only enforces the `ai.rever.boss.plugin.*` contract** (#762): `BinaryCompatibilityValidator` was disabling plugins over method/field references into *bundled* third-party libraries — e.g. terminal-tab bundles ktor (compiled vs kotlinx-serialization 1.9.x) which calls `IoStreamsKt.encodeToSink/decodeFromSource`; the host runs kotlinx-serialization-json-io 1.11.0 (different signature) and the whole plugin was disabled even though that path may never run. Class resolution was already scoped to `ai.rever.boss.plugin.*` owners; member (method/field) resolution now mirrors that. Third-party signature drift degrades at the actual call site at runtime (handled by the plugin's own error handling) instead of at load.
- **Pin ktor + kotlinx-serialization to the MCP SDK stack**: the v9.2.2 auto-bump to ktor 3.5.0 + serialization 1.11.0 broke the bundled MCP server in terminal-tab — ktor 3.5.0's `sse()` signature differs from what mcp-sdk 0.8.3 expects, and `encodeToSink`/`decodeFromSource` were removed post-serialization 1.9.x. Reverted to the prior known-good stack (ktor 3.4.3, serialization 1.9.0); CI/release workflows targeting ktor 3.5+ wait for MCP SDK support (tracked in #772 / kshivang/BossTerm#269).

## Bug Fixes

- **plugins**: `copyBundledPluginsLocal` scans `../../boss_plugins/...` (correct plural), so the app's bundled plugin copies stay in sync with local builds and the recurring "Source directory not found" warnings stop. (#754)
- **plugins**: `scheduleBackgroundUpdateCheck` guards the GitHub download with `isNewerVersion(latest, installed)`, so a locally built plugin ahead of the published release is no longer downgraded and its jar is no longer deleted by `downloadSystemPluginFromGitHub`. (#754)
- **validator**: `BinaryCompatibilityValidator` scopes member resolution (method/field) to `ai.rever.boss.plugin.*` owners, matching the existing class-resolution scope. Signature drift in a plugin's *bundled* third-party libraries (e.g. ktor → kotlinx-serialization) no longer disables the plugin at load. (#762)
- **validator**: Per-class loop guard prevents disabling a plugin over optional third-party adapter classes (e.g. kotlin-logging's logback adapter referencing classes not in the host classpath). (#762)

## Dependencies

- ktor pinned to **3.4.3** (reverted from 3.5.0) to match the MCP Kotlin SDK ≤0.12.0 target. (#762)
- kotlinx-serialization pinned to **1.9.0** (reverted from 1.11.0); `encodeToSink`/`decodeFromSource` were removed post-1.9.x and broke the bundled MCP server in terminal-tab. (#762)
