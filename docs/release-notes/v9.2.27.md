# BOSS v9.2.27

**Release Date:** 2026-07-03

Stability release triggered by the Toolbox 1.8.4 incident: the plugin updater now gates store updates on `minBossVersion` before installing, the host's own `McpServerController` interface carries `setPort`, and startup plugin passes no longer race. Also makes `.deb` desktop-menu integration best-effort so headless installs no longer half-fail.

## Highlights

- **Plugin updates gated on host compatibility**: The updater now rejects store candidates whose `minBossVersion` exceeds the running host *before* replacing the working jar, so an incompatible update (like Toolbox 1.8.4 on BOSS 9.2.25) is surfaced as an incompatible notice instead of bricking the plugin (#829).
- **`McpServerController.setPort` in the host interface**: The host's `plugins/plugin-api-core` copy of `McpServerController` now carries `setPort`, so plugins that reference it (Toolbox 1.8.4, terminal-tab 2.5.9) load without failing `BinaryCompatibilityValidator` (#827).
- **Headless `.deb` installs succeed**: `.deb` maintainer scripts no longer abort under `set -e` when `xdg-*` desktop-integration tools fail on headless systems, so servers/containers/CI runners install and remove cleanly (#828).

## Bug Fixes

- **plugins**: Consolidated hardening from the Toolbox 1.8.4 incident. `PluginUpdateManager` rejects store candidates whose `minBossVersion` exceeds the injected `hostBossVersion` (mirroring the IPC gate, reported as `IncompatibleNotice`, failing open on blank/unparseable versions so the loader stays the backstop); `IncompatibleNotice` gains `requiredBossVersion`/`hostBossVersion` so it names the real reason; `RemotePluginRepository`/`PluginStoreClient` actually map `minBossVersion` from the store response (previously silently defaulted); `PluginStoreSetup` injects `AppVersion.currentVersionString()`; and the manual publish scripts read `minBossVersion` from the jar's `plugin.json` instead of hardcoding `1.0.0`. Also sequences the two startup plugin passes so the directory scan (`loadExternalPlugins`) no longer races the persisted-plugin pass — it joins the persisted load first, skips jars the manager already tracks, treats stale-version duplicates as info-level skips, and stops force-loading plugins whose persisted entry is disabled — eliminating the spurious "Plugin already loaded" failures ("total=28, successful=11, failed=17") seen every launch. Backed by new desktop tests covering reject/offer/fail-open paths (#829).
- **api**: Sync `McpServerController.setPort` into the host's `plugins/plugin-api-core`. `boss-plugin-api` gained `setPort` in 1.0.55 and terminal-tab 2.5.9 implements it, but the host's own interface — what the plugin parent classloader serves at runtime — was never updated, so Toolbox 1.8.4 failed `BinaryCompatibilityValidator` with "setPort((I)V): method not found" and was disabled at load even on BOSS 9.2.26. The validator statically scans bytecode at load time, so no runtime try/catch guard helps; adding the method to the host interface is the whole fix (no host code implements the interface) (#827).
- **packaging**: Make `.deb` desktop-menu integration best-effort for headless installs. jpackage's generated `postinst`/`prerm` run `xdg-desktop-menu` under `set -e`; on headless systems it exits 3 ("No writable system menu directory found"), aborting `dpkg` configure/remove and leaving the package half-installed (every DEB job in the BossConsole-Releases "Test Install Scripts" workflow was red). The `fixLinuxDesktopFile` post-packaging task now appends an `|| echo …` fallback to the `xdg-*` lines in `DEBIAN/postinst`/`prerm` so missing desktop integration degrades to a warning. Verified in a headless `debian:12` container against the real v9.2.25 deb, including self-healing over a broken half-install (#828).

## Improvements

- **packaging**: Tighten the deb xdg patch per review. The maintainer-script regex is anchored with optional leading whitespace and broadened from `xdg-desktop-menu` to the whole `xdg-*` class (`xdg-icon-resource`, `xdg-mime`, …) so a future jpackage that indents or reorders these commands can't silently reintroduce the headless-install failure; the deb is only repacked when the maintainer-script text actually changed; and the stale "Repacked … with StartupWMClass" log message is generalized now that the task applies three kinds of desktop-integration fixes (#830).
