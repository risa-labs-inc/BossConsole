# BOSS v9.2.49

**Release Date:** 2026-07-20

Fixes the release-publishing pipeline that made v9.2.48 invisible to the in-app updater: resumable uploads now slice chunks byte-accurately (so any TUS resume offset is valid), and a post-publish assertion fails loudly if the `app_releases` row that `latest-release` reads wasn't written.

## Highlights

- **Byte-accurate TUS resumable uploads**: Chunks were sliced by chunk *index* (`dd bs=CHUNK_SIZE skip=offset/CHUNK_SIZE`), which requires the resume offset to land on a `CHUNK_SIZE` boundary. A partially-applied PATCH leaves the TUS server on a non-boundary offset, so the resync aborted with `misaligned offset` instead of resuming — turning a transient large-asset blip into a hard publish failure. Slicing now uses portable `tail | head`, making any server-reported resume offset valid (#877, #878).
- **Post-publish assertion**: After the upsert, the `app_releases` row is read back and the publish fails loudly if it's absent — so a silent no-op upsert or an upload that aborts before the DB insert can no longer leave a release on GitHub and in the bucket but invisible to clients (which is exactly what happened to v9.2.48) (#877, #878).

## Bug Fixes

- **release**: Byte-accurate TUS slicing + post-publish assertion. `slice_chunk()` slices byte-exactly via `tail | head`, removing the two alignment gates and the `dd` chunk-index math so partial-PATCH resume offsets resume instead of aborting. The publish step now reads the `app_releases` row back after the upsert and fails if it's missing. Immediate remediation was done out-of-band: `sync-release` was re-dispatched for v9.2.48 so `latest-release` returns it again (#877, #878).

## Improvements

- **release**: Hardened `slice_chunk` to be SIGPIPE-safe and testable — `head` closing the pipe after `LEN` bytes killed `tail` with 141, which `pipefail` would have surfaced as a failure (masked only by the `|| exit 1` call site); it now swallows `tail`'s status and asserts the output size, and both call sites use `|| return 1` so a genuinely short/truncated read fails loudly. Extracted `CHUNK_SIZE` + `slice_chunk` into `lib/tus-slice.sh` with a `scripts/test/test-tus-slice.sh` that runs the real function under `set -euo pipefail`, asserting both bytes and exit status at aligned / unaligned-non-final / short-final offsets — wired into `build.yml` as a new 🐚 Shell Script Tests job. The post-publish check uses `-G` + `--data-urlencode` and `jq -e 'type=="array" and length>0'` so `+build.meta` versions filter correctly and a stray object body can't false-pass (#877, #878).
