# BOSS v9.2.10

**Release Date:** 2026-06-05

Makes `BrowserService` the single browser entry point for plugins (opt-in managed profiles, JS execution, auth seeding), adds a dismissible update dialog with persistent per-version dismissal, and reconciles duplicate plugin JARs so the newest version always loads.

## Highlights

- **Central browser provider** (#775): Browser automation folds into the existing `BrowserService` (replacing the separate `RpaBrowserProvider`), giving plugins one browser entry point with opt-in managed profiles, JavaScript execution, and auth seeding. General browser tabs are unaffected.
- **Dismissible update dialog** (#773): A new-version detection now shows a dialog (alongside the existing banner) that can be dismissed per-version — automatic checks stay quiet for that version, while a newer release or a manual "Check for Updates" prompts again.
- **Duplicate plugin JAR reconciliation** (#774): A new `PluginJarReconciler` groups JARs by manifest plugin ID, keeps the highest version, and removes the rest, so a stale JAR can no longer shadow a newer one at startup ("Plugin already loaded").

## Features

- **browser**: `BrowserService` becomes the central browser provider — `seedProfile` / `deleteProfile` / `listProfiles`, `BrowserHandle.loadUrlAndWait` / `executeJavaScript`, `BrowserIntegration.navigate`, `ActiveTabsProvider.createBrowserTabInRightSplit`, and `SplitViewOperations.openTab(tabInfo)`. `BrowserConfig` gains managed-profile options (`profileName` / `ephemeralProfile` / `auth`) as post-construction properties to stay binary-compatible with pre-built plugins. `BrowserServiceImpl` absorbs the managed-profile subsystem (cookie/header auth seeding, ephemeral + persistent named profiles, per-profile fencing, LRU disk-cap eviction, atomic `meta.json`, orphan sweep); `RpaBrowserProvider` and `PluginContext.rpaBrowserProvider` are removed. Mirrors boss-plugin-api 1.0.42. (#775)
- **updater**: A dismissible update dialog prompts the user when a new version is detected, rendered in exactly one window via reactive ownership over `WindowManager.windows` (no duplicates in multi-window setups; ownership transfers if the owning window closes). Dismissal — via "Later", Esc/scrim, or the banner's "Dismiss" — persists the version in `~/.boss/update-settings.json`; a newer version or a manual force check re-prompts. (#773)

## Bug Fixes

- **browser**: Fixed a managed-profile leak/deadlock where a throw after `newBrowser()` (overscroll settings, popup-POST replay, handle construction, named-profile meta persist) returned `null` without releasing the managed profile, leaving a named profile's fence locked for the process lifetime and leaking ephemeral profiles. (#775)
- **browser**: `DesktopBrowserAccessor` now keys the browser-integration cache by tab id, fixing a stale integration being returned for a different tab. (#775)
- **plugins**: Stale plugin JARs no longer shadow a newer version at startup. The directory scan previously loaded whichever JAR the OS listed first (different writers use `pluginId-version.jar`, `pluginId_version.jar`, or arbitrary GitHub asset names), so an older version could win and block the rest as "Plugin already loaded". `PluginJarReconciler` runs at the startup funnel and after a successful update, keeping the highest version and repointing `installed.json` at it (microkernel runtime and non-plugin JARs untouched). (#774)
- **updater**: `dismissVersion()` now clears the dialog/banner UI before persisting to disk, so on a slow disk the prompt no longer stays visible and re-tappable until the `Dispatchers.IO` write completes. (#773)
