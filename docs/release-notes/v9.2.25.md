# BOSS v9.2.25

**Release Date:** 2026-07-03

Fixes a startup race where "Last Session" restore ran before the dynamic plugins that own the browser/terminal/editor tab types had registered, causing restored tabs (and project-selection panels like CodeBase/RunConfigurations) to be silently dropped.

## Highlights

- **Reliable session restore at startup**: Workspace restore now waits (bounded to 15s, listener-driven) for the tab types and panels its layout needs before applying — so restored browser/terminal/editor tabs and project-open panels no longer vanish when dynamic plugin loading loses the race (#821).

## Bug Fixes

- **workspace**: Gate session restore on plugin tab-type registration. At startup, Last Session restore and dynamic plugin loading kicked off concurrently with no synchronization; restore always won (the workspace flow emits instantly while plugin install is disk/network-bound), so `TabRegistry.createTabComponent` found no factory and `addTab` silently dropped every restored tab — the same race also dropped `PanelEventBus` open events (CodeBase/RunConfigurations on project selection). A new bounded `awaitRegistryCondition()` (shared by `TabRegistry`/`PanelRegistry` consumers) makes `applyWorkspace()` wait for the tab types its layout needs before `clearAllPanels()` (the preserved-state fast path stays instant and the wait is a no-op once plugins are loaded), the panel-open collector waits for its panel to register, the fresh-install fallback no longer marks deep-link handlers ready while a restore is in flight, and `addTab` now logs a warning when it drops a tab for a missing factory. The Last Session `applyWorkspace` is wrapped in try/catch (rethrowing cancellation) so a throwing restore can't leave URL/file/terminal/workspace handlers unmarked for the session, and `tabTypeIdFor` becomes the single source of truth for persisted-type → tab type id so the pre-apply wait and tab construction can't drift apart (#821)
