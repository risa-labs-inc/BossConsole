# BOSS v9.2.36

**Release Date:** 2026-07-08

Turns the plugin system into a full platform: a hot-swappable, runtime-updatable API layer (process-wide `ApiClassLoader` with `minApiVersion` gates and store-only API distribution), five plugin-extensible host UI registries behind crash boundaries, and a Supabase-backed system-plugins manifest — plus a fix for the Toolbox setup wizard re-popping on every restart (#847, #846).

## Highlights

- **Hot-swappable API layer**: The `boss-plugin-api` surface no longer compiles into the host binary — plugin-api-core drops its entire API source tree (~9,300 lines) and the API classes are loaded at runtime through a process-wide `ApiClassLoader`, so the API can be updated (and hot-swapped) independently of the host. `minApiVersion` gates reject plugins built against a newer API than the running host, and API packages are distributed store-only with local api-package filtering kept just for compilation (#847).
- **Five plugin-extensible UI registries**: Plugins can now contribute into the status bar, panel ⋮ menu, keyboard shortcuts, settings pages, and deep-link actions via dedicated registries (`StatusBarRegistryImpl`, `PanelMenuRegistryImpl`, `PluginShortcutRegistryImpl`, `SettingsPageRegistryImpl`, `DeepLinkActionRegistryImpl`). Each contribution renders inside a crash boundary (`PluginExtensionBoundary`) with RBAC/ownership gating (`AccessGatedRegistry`, `PluginOwner`), so a misbehaving plugin can't take down the host UI (#847).
- **Supabase-backed system-plugins manifest**: A new `SystemPluginManifestService` resolves the set of system plugins from a remote Supabase manifest (migration `20260708000000_system_plugins_manifest.sql`), letting the bundled/system plugin set be managed server-side instead of hard-coded in the app (#847).
- **Toolbox wizard no longer re-pops on restart**: The setup-wizard check raced asynchronous startup plugin loading and, when it won, saw an empty registry and re-showed the wizard even though it was already completed. `DefaultPlugin.awaitInitialPluginLoad()` now joins both startup load jobs (30s cap) before trusting the installed-plugin count, so the wizard shows only on genuine first runs (#846).

## Features

- **plugins**: runtime-updatable, hot-swappable API layer — a process-wide `ApiClassLoader`, `minApiVersion` gates, store-only API distribution, and local api-package filtering for compilation; plugin-api-core no longer ships the API source in the host binary (#847)
- **plugins**: five plugin-extensible UI registries — status bar, panel ⋮ menu, keyboard shortcuts, settings pages, and deep-link actions — each behind a `PluginExtensionBoundary` crash boundary with RBAC/ownership gating via `AccessGatedRegistry`/`PluginOwner` (#847)
- **plugins**: Supabase-backed system-plugins manifest (`SystemPluginManifestService`, migration `20260708000000_system_plugins_manifest.sql`) so the system plugin set is resolved remotely instead of hard-coded (#847)

## Bug Fixes

- **plugins**: Toolbox setup wizard no longer pops on every restart — the wizard check awaits `DefaultPlugin.awaitInitialPluginLoad()` (joining the persisted + external-scan startup jobs, 30s cap) before trusting the installed-plugin count, fixing a race where an empty registry looked like "no plugins installed" (#846)
