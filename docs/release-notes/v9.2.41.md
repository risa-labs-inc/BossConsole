# BOSS v9.2.41

**Release Date:** 2026-07-11

Recovers from Supabase session-refresh failures instead of crash-looping: a stuck `RefreshFailure` session now runs a single-flight recovery loop with exponential backoff rather than letting supabase-kt's Realtime reconnect throw `TokenExpiredException` into the crash dialog every ~40s (#855).

## Highlights

- **Session refresh no longer crash-loops the app**: `CoreAuthService` now handles `SessionStatus.RefreshFailure` — previously only `Authenticated`/`NotAuthenticated` were handled, so after a failed token refresh the app sat on an expired session while supabase-kt's Realtime reconnect threw `TokenExpiredException` into library-internal coroutines every ~40s, each hitting `CrashHandler` and popping the "BOSS Has Crashed" dialog (whose dismissal exits the app) (#855).
- **Recover, or fall back to login — never a zombie session**: A single-flight recovery loop retries `refreshCurrentSession` with exponential backoff (10s→60s). Transient network/5xx/Cloudflare failures (plus 408/429) keep retrying without signing out, while a hard 4xx rejection of the refresh token (revoked or rotated by another instance) does a full cleanup and drops the user on the login screen instead of a stuck expired state (#855).

## Bug Fixes

- **auth**: recover from Supabase session refresh failures instead of crash-looping — `CoreAuthService` handles `SessionStatus.RefreshFailure` and runs a single-flight, exponential-backoff (10s→60s) recovery loop; a `SessionRecoveryPolicy` makes the testable retry-vs-clear decision (transient network/5xx/Cloudflare/408/429 retry; remaining 4xx clear the session via `clearUnrecoverableSession()` — local `auth.clearSession()` first, then `SessionManager.clearSession()`). A first-status-after-startup `RefreshFailure` now resolves the session and surfaces **Offline** instead of hanging on the Loading spinner. `signOut()` `cancelAndJoin()`s any in-flight recovery so it can't race sign-out, `recoveryJob` is `@Volatile`, and `CrashHandler` now classifies `TokenExpiredException` (including in cause chains) as ignorable since it is a recoverable background auth failure, not an app crash (#855).
