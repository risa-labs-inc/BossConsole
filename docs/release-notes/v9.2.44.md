# BOSS v9.2.44

**Release Date:** 2026-07-12

Adds a **System Default** option to the terminal link-open dialog, so URLs and files clicked in a terminal can be routed to the OS default handler as an explicit, rememberable choice.

## Highlights

- **System Default link-open mode**: `TerminalLinkOpenMode` gains `SYSTEM_DEFAULT` — a new option card in `TerminalLinkOpenDialog` and an entry in the Terminal Links settings dropdown. URLs open via the existing `openUrlInBrowser` expect/actual; files open via a new `openFileWithSystemDefault` expect/actual backed by `FileSystemUtils.openFile` (Finder/Preview, not just a browser). Pairs with BossTerm #334, which forwards CLI-originated `open`/`xdg-open`/`$BROWSER` requests into the same link-click pipeline — those now show the dialog with System Default as a selectable, persistable choice instead of silently opening the OS default (#863).

## Features

- **terminal-links**: add a **System Default** option to the link-open dialog and Terminal Links settings — URLs route through `openUrlInBrowser` and files through a new `openFileWithSystemDefault` (`FileSystemUtils.openFile`), with a mode-agnostic `AutoMirrored.Outlined.Launch` icon distinct from Existing Split's `OpenInNew`. Pairs with BossTerm #334 so CLI-originated `open`/`xdg-open`/`$BROWSER` requests from terminal sessions flow through the same link-click dialog. Adds `TerminalLinkSettingsSerializationTest` guarding that every mode round-trips through persisted JSON (`SYSTEM_DEFAULT` decodes by name, missing fields fall back to defaults) against enum renames/reordering, plus log-and-noop actuals for `openFileWithSystemDefault` on the disabled android/ios/wasmJs targets to keep the expect/actual pattern uniform (#863).
