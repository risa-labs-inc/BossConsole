# BOSS v9.2.11

**Release Date:** 2026-06-06

Fixes a destructive startup cycle between the `boss-plugin-terminal` and `boss-plugin-terminal-tab` system plugins by matching plugin JARs on their manifest plugin ID instead of a filename prefix.

## Highlights

- **No more terminal-plugin tug-of-war at startup** (#777): Because `boss-plugin-terminal` is a filename prefix of `boss-plugin-terminal-tab`, the two cleanup/version-check sites in `PluginStoreSetup` kept deleting each other's JARs in a loop on launch. Matching by manifest plugin ID stops the cycle.

## Bug Fixes

- **plugins**: `PluginStoreSetup` no longer deletes a sibling system plugin's JAR whose artifact name shares a prefix. `copyBundledPluginsToPluginDir` (which derived a prefix via a double `substringBeforeLast("-")`) and `ensureSystemPluginsInstalled`'s post-download cleanup (`startsWith(artifactPrefix)`) both treated `boss-plugin-terminal` as matching `boss-plugin-terminal-tab-*.jar`, producing a destructive cycle: terminal-tab's bundled copy deleted terminal's JAR as an "old version", then terminal was re-downloaded and its cleanup deleted terminal-tab's JAR right back. Both sites now match candidates by reading each JAR's manifest plugin ID (via the existing `readPluginManifest` helper, which returns `null` for unreadable/non-plugin JARs — those are never touched). The `downloadOnly` lookup keeps its prefix match, since it only runs for the microkernel runtime, whose prefix has no collision candidates. (#777)
