# BOSS v9.2.43

**Release Date:** 2026-07-12

Fixes crash reporting for end users by routing reports through a server-side proxy that holds the GitHub token and files each issue in the crashing plugin's own repo, and makes the dashboard header lay out gracefully in narrow windows.

## Highlights

- **Crash reports work without a local GitHub token**: crash reporting failed for end users with "GitHub authentication required" because the client created GitHub issues directly, needing a token that no user machine has. A new server-side crash-report Edge Function now holds `GITHUB_TOKEN` and files the issue in the crashing plugin's own repo (`system_plugins.github_repo`, then `plugins.homepage_url`), falling back to `BossConsole-Releases` for host crashes; the client submits through this proxy first and only uses direct GitHub as a dev-machine fallback (#862).
- **Crashes are attributed to the plugin that caused them**: `CrashHandler` walks the stack root-cause-first and blames the first frame whose class was defined by a `PluginClassLoader`, so a plugin's crash lands as an issue in that plugin's repo instead of the host's.
- **Dashboard header adapts to narrow windows**: the stats card no longer overlaps the welcome title and stat labels no longer wrap letter-by-letter — under 700dp the stats card stacks below the title, and at very narrow widths it scrolls horizontally like the rest of the dashboard instead of overflowing (#861).

## Bug Fixes

- **crash-reporting**: route crash reports through a server-side proxy so reporting works key-free for every user. A new crash-report Edge Function holds `GITHUB_TOKEN` server-side and files the issue in the crashing plugin's repo (`system_plugins.github_repo`, then `plugins.homepage_url`) or `BossConsole-Releases` for host crashes, with signature dedup and a fallback to the default repo if the plugin repo rejects it. Dedup scans the 100 newest open crash-report issues via the issues **list** API rather than `/search/issues` (the search index lags seconds-to-minutes, so a crash loop would file a duplicate per crash; the list endpoint is read-after-write consistent). `CrashHandler` attributes each crash to a plugin via the first `PluginClassLoader`-defined stack frame (the loader now keeps a weak process-wide instance registry), and `CrashReportService` submits via the proxy first, keeping the direct GitHub API only as a dev-machine fallback (#862).
- **dashboard**: make the dashboard header responsive to narrow windows — `DashboardHeader` stacks the stats card below the title under 700dp via `BoxWithConstraints`, gives the title `weight(1f)` in the wide layout, pins stat texts to a single line, and lets the fixed 3-item stats row scroll horizontally at very narrow widths instead of overflowing (#861).

## Improvements

- **crash-reporting**: harden the crash-report function — add `[functions.crash-report]` (`verify_jwt=false`) to `supabase/config.toml` so a redeploy can't silently flip to gateway JWT checks and 401 every report; require the `apikey`/`Bearer` header to match `SUPABASE_ANON_KEY` to block anonymous drive-by issue creation; add a per-IP sliding-window rate limit (20/hour) consumed only after auth and validation so malformed requests can't burn honest clients' budget; drop CORS entirely since the desktop client isn't subject to it; and tail-truncate plugin labels over GitHub's 50-char cap so reverse-DNS ids don't collide on their shared prefix (#862).
