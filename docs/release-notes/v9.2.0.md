# BOSS v9.2.0

**Release Date:** 2026-05-29

Pivots the terminal out-of-process strategy: the host classloader drops its `bossterm-compose` dependency entirely, and the terminal-tab plugin bundles `bossterm-compose` privately inside its own JAR. Plugin-side BossTerm version bumps now trigger plugin-only re-releases instead of host re-releases, leaving Compose Multiplatform as the only shared coupling.

## Highlights

- **Host drops `bossterm-compose`**: `composeApp/build.gradle.kts` removes `runtimeOnly(libs.bossterm.compose)` and `gradle/libs.versions.toml` drops the `bossterm` version pin + the `bossterm-compose` library coordinate. Confirmed via `./gradlew :composeApp:dependencies --configuration desktopRuntimeClasspath` â€” only `bosseditor-compose-desktop` remains under `com.risaboss`. (#742)
- **Plugin bundles BossTerm privately**: The plugin's `@Composable` functions execute inside the host's Compose runtime via parent-classloader delegation; the BossTerm classloader lives inside `boss-plugin-terminal-tab` v2.1.0+ (and `boss-plugin-terminal` v2.1.0+ for the sidebar variant). End-to-end verified â€” terminal tab opens with full BossTerm UX (colors, mouse, selection, scrollback, type-ahead). (#742)
- **Renderer-rewrite path retired from `main`**: The dormant Phase A/B/C scaffolding from v9.1.11's PR #733 is removed â€” 11 files under `composeApp/src/desktopMain/.../terminal/render/` (`GridState`, `KeyEventEncoder`, `LoopbackTerminalServer`, `StubTerminalGridSource`, `TerminalGridClient`, `TerminalGridConnection`, `TerminalGridRenderer`, `TerminalGridSource`, `TerminalRenderDevWindow`, `TerminalRenderFlag`, `ThemePalette`) plus 4 tests and the gated dev-preview mount in `main.kt`. IPC v1.1.0 proto additions and `boss-ipc:1.1.0` are kept (additive, cheap). (#742)
- **`BinaryCompatibilityValidator` tightening**: Soft-fails references into `ai.rever.boss.plugin.runtime.*` (OOP child-JVM-only classes legitimately referenced from in-process register paths that only execute remotely), and stops flagging interface references to `Object` methods (`toString` / `equals` / `hashCode` / `getClass`) â€” the prior logic mis-flagged cases like JNA's `CallbackProxy -> java.util.List.toString()` because `List.getMethod("toString")` doesn't surface inherited `Object` methods via reflection. (#742)

## Features

- **terminal**: Terminal plugin owns its `bossterm-compose` dependency. Plugin classloader provides the BossTerm classes; host has no compile-time or runtime BossTerm coupling. (#742)

## Improvements

- **plugin-loader**: `BinaryCompatibilityValidator` extracts an `isSoftFailReference` helper, logs on skip for diagnosability, uses `Any::class.java` instead of `java.lang.Object::class.java`, and exposes helpers as `internal` for test reach. (#742)
- **build**: Drop the v9.1.12 version anchor from the bossterm-not-a-host-dep comment in `composeApp/build.gradle.kts` so the rationale doesn't rot. (#742)

## Bug Fixes

- **plugin-loader**: Stop incorrectly flagging `Object`-method calls dispatched through interface receivers â€” e.g. JNA's `CallbackProxy -> java.util.List.toString()` is now correctly treated as a valid reference. (#742)

## Tests

- **plugin-loader**: New `BinaryCompatibilityValidatorTest` pins the two behavioural deltas: runtime-package soft-fail, and the `Object`-method-on-interface case that surfaced as a false positive when JNA's `CallbackProxy` referenced `java.util.List.toString()`. (#742)
