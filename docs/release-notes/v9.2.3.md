# BOSS v9.2.3

**Release Date:** 2026-06-01

Two split-view and run-config focus/path fixes: clicked splits no longer lose focus when another split prints output (root cause was both AWT-backed children swallowing Compose focus events *and* `buildList` recomposition calling `setActivePanel` on every output line), and auto-detected run configs now `cd` to the script's project root so `./gradlew :module:run` resolves when the active workspace differs from the script's location.

## Highlights

- **Clicked split keeps focus when another split emits output** (#750, #753): AWT-backed children (JxBrowser, native components) don't bubble Compose focus events to the parent, so `onFocusChanged` never fired for those splits and `setActivePanel` was never called — the output-producing split stayed "active" and re-asserted focus on every recomposition. Reproducible even with two plain terminals because the tab context-menu items were built with `buildList { ... }` (which runs on every tab-bar recomposition, including on every terminal output line) and the list called `trackTabInteraction(currentPanelId, ...) → setActivePanel()`. Fix is two-pronged: a `pointerInput` on the panel observes presses via `PointerEventPass.Initial` without consuming them (so children still receive the event) and calls `setActivePanel` for the clicked panel, and the composition-time `trackTabInteraction` call is removed because left-click (tab `onClick`) and right-click (panel `pointerInput`) already handle activation.
- **Run configs `cd` to the script's project root, not the workspace path** (#752): auto-detected run configs generated a relative command (e.g. `./gradlew :module:run`) against the script's own project root but set `workingDirectory` to the scanned workspace path — when the active workspace differs from the script's location, the command ran where `./gradlew` doesn't exist (effectively `$HOME`) and the run silently broke. `DesktopMainFunctionDetector` now derives `workingDirectory` from the script's project root via the existing `findProjectRoot()` so the `cd` target matches the command and is never blank.

## Bug Fixes

- **panel**: Split-view focus stealing when one split produced output is fixed — `BossMainWindowPanel` adds a `PointerEventPass.Initial` press observer so clicks into AWT-backed splits still mark the panel active, and the `buildList`-driven `trackTabInteraction(currentPanelId, ...)` call is removed from the tab-bar so a split merely emitting output no longer re-marks itself active on every recomposition. (#750, #753)
- **run**: `DesktopMainFunctionDetector` sets `workingDirectory` to the script's project root via `findProjectRoot()` rather than the scanned workspace path, so `./gradlew :module:run` resolves correctly when the active workspace differs from the script's location. (#752)
