# BOSS v9.2.52

**Release Date:** 2026-07-20

Opens `.ipynb` files in a dedicated Jupyter notebook tab rendered by the jupyter-notebook plugin, and unbreaks the Release Build with config-cache-safe macOS signing tasks plus a Windows-ARM64 IPC test guard.

## Highlights

- **Jupyter notebook tabs**: `.ipynb` files opened from the file tree, bookmarks, or **New Tab → Jupyter** now route to a `JupyterTabInfo` tab (typeId `jupyter`) served by the jupyter-notebook plugin — with an optional notebook-name field in the New Tab dialog, workspace save/restore support, and a graceful fall back to the code editor when the plugin isn't installed (#854).
- **Release Build unbroken**: The macOS signing tasks and the Windows-ARM64 IPC test dependency both broke the 🚀 Release Build under the configuration cache; both are now fixed so signed multi-platform builds publish again (#888).

## Features

- **tabs**: Open `.ipynb` files in a Jupyter notebook tab. `JupyterTabInfo` (typeId `jupyter`, added to plugin-api-core) is rendered by the jupyter-notebook plugin; `.ipynb` routing from the file tree / bookmarks / **New Tab → Jupyter** threads `filePath` through the tab handlers, gates on the plugin being registered (falling back to the code editor otherwise), and persists/restores across workspace save/restore. New Tab → Jupyter offers an optional notebook-name field (blank → "Notebook") that flows to the tab title, split copies get a fresh id, `updateTitle` now applies the plugin's rename updates, and a `JupyterTabInfo.create()` factory with a Windows-safe basename dedupes the call sites. Adds `JupyterTabInfoTest` (#854).
- **tabs**: Add **Reveal in Finder** to the tab right-click menu. Right-clicking a file-backed tab (editor or notebook) offers a platform-aware reveal-in-file-manager action (Finder / Explorer / folder) via a new `commonMain` `revealInFileManager` util that the tab context menu reads reflectively for dynamic plugin tabs (#854).

## Bug Fixes

- **build**: Unbreak the 🚀 Release Build on macOS and Windows-ARM64, both in `composeApp/build.gradle.kts`. `signPty4jBinaries` and `extractCLIToAppResources` captured script-level values and called `project.layout`/`project.file` inside their `onlyIf`/`doLast` lambdas, which the configuration cache can't serialize (`Cannot invoke … because this$0 is null`); both now capture everything as task-local values at configuration time while keeping providers lazy. On Windows-ARM64 the `desktopTest` dependency on `plugin-api-ipc` used a type-safe accessor for a module `settings.gradle.kts` excludes there (no protoc binary), failing script compilation — the dependency is now resolved by path behind a `findProject` guard, and `SkipListDriftTest.kt` is excluded from `desktopTest` when the module is absent (the drift check still runs on every other platform) (#888).

## Improvements

- **build**: Skip macOS signing when the Developer ID cert is absent, so local builds produce an unsigned distributable instead of failing `createDistributable`; CI (cert present) still signs (#854).
