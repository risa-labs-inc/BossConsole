# BOSS v9.2.47

**Release Date:** 2026-07-19

Signs store plugins and verifies those signatures at plugin load time — the choke point every install path funnels through — so a rewritten store row or storage object can no longer smuggle a different JAR onto a user's machine. Also removes ~2,850+ lines of dead code left behind by earlier plugin/RPA migrations.

## Highlights

- **Load-time plugin signature verification (covers every install path)**: `#871` wired verification into `RemotePluginRepository`, but the Toolbox (plugin-manager) uses its own `downloadFromStore` and never hit that path — so store installs/updates, the dominant surface, were checksum-only. Verification now lives at the load choke point: `DynamicPluginLoader.loadPlugin` verifies a `<jar>.sig` sidecar over the anchor `pluginId|version|sha256` (manifest identity + computed hash + sidecar signature) against the pinned key. Present-but-invalid signatures reject with `PluginSignatureException`; absent signatures warn (dev side-loads exempt via `boss.dev.mode`, hard-fail once enforcement is enabled) (#875).
- **Signed store plugins**: The store's publish function signs each version's server-computed sha256 (RSASSA-PKCS1-v1_5/SHA-256 over the lowercase hex digest) with a key held as a Supabase secret; the download route returns the signature and the host verifies it against the pinned store public key after its existing local checksum check — binding signature → hash → bytes. A strict `verifyDigestSignature` path avoids the standard-jarsigner fallback that would otherwise accept any self-signed JAR (#871).
- **Dead-code cleanup**: Removes the orphaned RPA panel subtree and ~2,850 more lines with zero references across BossConsole, boss_plugins, BossTerm, and BossEditor — with the genuinely live browser-integration pieces extracted verbatim so there is no behavior change (#870, #869).

## Features

- **security**: Verify plugin signatures at load time, covering all install paths — verification moves to `DynamicPluginLoader.loadPlugin` (the single point every install/update path funnels through), `PluginSignatureVerifier`/`PluginStoreTrust`/`PluginSignatureEnforcement` relocate into `plugin-loader`, a shared `FileHashing.sha256` replaces the two byte-identical hash utils, blank signatures are treated as absent (never writing a zero-byte `.sig`), and the publish route asserts the JAR manifest's `pluginId`/`version` match the row being signed so the download-time and load-time anchors can't diverge. Load-time enforcement stays gated by #872 (#875).
- **security**: Sign store plugins and verify signatures at install — adds the publish-time signing util, a `plugin_versions.signature` migration, PEM public-key support, a DB-only backfill script (with a pager-termination fix for skip-only rows), and verifier unit tests. Invalid signatures hard-fail the install; missing ones warn-and-allow until the backfill completes (#871).

## Improvements

- **chore**: Remove the dead RPA panel subtree and extract the live browser bridge — the bundled RPA panels (`RpaEngine`, `LLMRpa`, `RpaRecorder`, `RpaActionExecutor`, `RpaEventCapture`) had no callers after #603 moved the feature to dynamic plugins; the live pieces (`BrowserIntegration`, the `BrowserAccessor`/`createFluckTabInfo`/`storeSplitViewState` expects, `FluckTabInfo`) move verbatim to `BrowserIntegration.kt` so imports and platform actuals are unchanged (#870).
- **chore**: Remove dead code identified by a workspace-wide audit — deletes ~2,850 lines with zero references across four repos: the RPA factory family orphaned by #603, platform actuals whose expects were removed in #63, RBAC DTOs superseded by the table-based schema migration, never-wired plugin-sandbox UI variants, and assorted born-dead helpers. No behavior change (#869).
