# BOSS v8.12.4

**Release Date**: October 2024

## Summary

This release introduces a comprehensive Role-Based Access Control (RBAC) system with Supabase Custom Claims integration, enabling secure role and permission management with JWT-based authentication for both magic link and passkey flows.

## Highlights

- Complete RBAC system with roles (user, admin) and 8 extensible permissions
- Supabase Auth Hooks inject role claims directly into JWT tokens
- Client-side role checking without additional API calls
- Secure role management with PostgreSQL RLS policies and SECURITY DEFINER functions
- Consistent JWT claims across both magic link and passkey authentication

## Features

### Role-Based Access Control (RBAC) System (#67)

**Database Layer:**
- PostgreSQL enums for roles (user, admin) and 8 permissions (can_create, can_read, can_update, can_delete, can_manage_users, can_manage_roles, can_manage_permissions, can_manage_settings)
- user_roles and role_permissions tables with Row Level Security (RLS) policies
- Auth hook function that injects custom claims (user_role, user_roles, is_admin) into JWT tokens
- Helper functions for role management and permission checking
- Auto-assignment of 'user' role on signup

**Kotlin Integration:**
- New RoleService for role management and JWT claim parsing
- RBACModels with AppRole, AppPermission enums and RoleClaims data class
- Updated AuthService with 7 new RBAC methods:
  - `getUserRoles()` - Fetch user's roles
  - `getUserPermissions()` - Fetch user's permissions
  - `assignRole()` - Assign role to user (admin only)
  - `removeRole()` - Remove role from user (admin only)
  - `hasRole()` - Check if user has specific role
  - `hasPermission()` - Check if user has specific permission
  - `isAdmin()` - Check if user is admin
- Integration with CoreAuthService, SessionManager, and PasskeySessionHandler

**Authentication Integration:**
- Magic link authentication includes role claims via auth hook
- Passkey authentication fetches roles and includes them in custom JWT
- Identical JWT claim structure across both authentication methods
- Client-side role checks use parsed JWT claims for immediate UI updates
- Server-side validation through JWT signature verification on all API requests

**Security Architecture:**
- Client-side JWT parsing for UI convenience (optimistic assumptions)
- Server-side JWT verification by Supabase on every API request (enforced guarantees)
- RLS policies use verified auth.jwt() claims
- Admin role can only be assigned by existing admins
- Extensible permission system for future plugin integration

## Bug Fixes

### Authentication (#67)

- **Fixed role assignment/removal API calls** - Changed from Edge Functions to PostgreSQL RPC using `client.postgrest.rpc()` instead of `client.functions.invoke()`
- **Fixed passkey JWT missing role claims** - Passkey authentication now includes user_role, user_roles, and is_admin claims matching magic link format

### Security (#67)

- **Replaced unsafe regex JSON parser** - Removed 54 lines of vulnerable regex code and replaced with kotlinx.serialization.json for secure JWT claim parsing
- **Fixed potential injection attack vector** - Proper JSON parsing prevents malformed JWT payloads from causing security issues

## Improvements

### Code Quality (#67)

- Removed printStackTrace() calls (replaced with println for error logging)
- Changed AppRole.values().toList() to AppRole.entries (modern Kotlin)
- Added comprehensive debug logging for RBAC JWT claims

### Documentation (#67)

- New RBAC_GUIDE.md with comprehensive usage examples
- Authentication Methods & RBAC section explaining magic link and passkey flows
- Critical security model documentation clarifying client vs server JWT verification
- JWT Security section with verification details and safe usage patterns
- Troubleshooting guide with debug log examples
