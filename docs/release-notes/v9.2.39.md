# BOSS v9.2.39

**Release Date:** 2026-07-10

Upgrades JxBrowser to 9.3.0 (Chromium 150) with a renewed license, which ships native macOS Touch ID / WebAuthn platform passkeys inside the branded engine â€” letting BOSS delete its ~1,337-line custom in-process implementation. Also hardens startup plugin loading so a background system-plugin update (e.g. a Toolbox self-update) landing mid-startup no longer leaves the plugin unloaded for the whole session (#852, #850).

## Highlights

- **JxBrowser 9.3.0 / Chromium 150 with renewed license**: Bumps `jxbrowser` 9.1.2 â†’ 9.3.0 (single source of truth in `libs.versions.toml`; engine default follows via `VersionConstants`) and updates the fallback license key to the renewed subscription â€” the previous key's update window ended mid-June 2026 and rejects 9.2.0+ with `InvalidLicenseException`. No API migration was needed (the only 9.3.0 breaking change, async `SelectMediaDeviceCallback`, is unused). Verified: desktop compile + `desktopTest` green, and a 9.3.0 engine boots on the published BOSS-branded `chromium-v9.3.0` binaries across all six platforms (#852).
- **Native macOS Touch ID passkeys, custom implementation removed**: Chromium 150 ships a native macOS Touch ID / WebAuthn platform authenticator, so the in-process implementation from #823 (WebAuthnBridge/Scripts/Origin, `MacPlatformPasskeys`, the `BossWebAuthn.swift` dylib and its Gradle pipeline, plus the explicit guava dependency) is removed â€” it was inert in practice anyway (its required Apple entitlement was never granted). Net effect: passkeys **improve** (native Touch ID works out of the box) with -1,337 lines to maintain. The branding pipeline now preserves the Developer ID identity + entitlements on re-sign, commits the Developer ID provisioning profile, and adds a keychain entitlement so Touch ID-verified passkeys created in the BOSS browser work (#852).
- **Startup survives background system-plugin updates**: A background update check (v1.8.7 Toolbox incident) that downloaded the new versioned JAR, rewrote `installed.json`, and deleted the old JAR mid-startup previously caused both startup passes to miss the plugin entirely â€” updated on disk but unloaded for the whole session (no updater, no store until restart). `loadPersistedPlugins` now re-resolves a missing jar path by manifest `pluginId` (`findRelocatedPluginJar`), `loadExternalPlugins` lists the plugins directory inside its coroutine (after the persisted pass) so the scan sees current state, and persistence is repointed at the jar that actually loaded (#850).

## Bug Fixes

- **plugin-platform**: survive background system-plugin updates during startup load â€” a background update that rewrites `installed.json` and swaps the JAR mid-startup no longer leaves the plugin unloaded; the persisted pass re-resolves relocated jars by `pluginId` and repoints persistence, and the external scan lists the directory after the persisted pass so it sees the fresh file (#850)
- **browser**: fix TCC `SIGABRT` (exit 134) during passkey ceremonies by adding `NSBluetoothAlways/PeripheralUsageDescription` to both the branded Chromium `Info.plist` (dev runs) and the outer packaged app (production) â€” Chromium 150's WebAuthn hybrid transport probes Bluetooth on ceremony start and macOS TCC hard-kills any responsible bundle lacking the key (#852)
- **browser**: clean up 9.3.0's renamed Chromium singleton lock file (`org.chromium.Chromium` â†’ `com.teamdev.Platinum`) in the startup stale-lock sweep so a leftover lock from a crashed session can't block profile reuse; the legacy name is kept for cached pre-9.3.0 engines (#852)
- **browser**: preserve framework symlinks in macOS engine zips (`zip -y -r`) so the code-signed "Chromium Framework.framework" bundle survives the round-trip â€” a flattened layout (tolerated by 9.1.2) broke 9.3.0's `ChromiumProcessMac.isInAppleSandbox` JNI load with `UnsatisfiedLinkError`; also roughly halves the macOS engine download by dropping the duplicated 238 MB binary (#852)
- **release**: fix Supabase TUS `409` when re-publishing an existing release asset (DELETE the object before the tus create) and the `app_releases` upsert `409` on re-publish (add `?on_conflict=app,version`) so re-publishing the same `(app, version)` succeeds (#852)

## Improvements

- **release**: fail the branding job on a silently mis-signed engine â€” after re-signing, assert `codesign --verify --deep --strict` and, when the Touch ID provisioning profile is present, require the embedded profile and `keychain-access-groups` entitlement to survive, so a stripped-entitlement engine no longer ships while CI reports success (#852)
- **release**: `latest-release` edge function now serves the `boss-chromium` engine via a new `&platform=` param (exact archive-name match, mutually exclusive with the apps' `download=`/`arch=` heuristics), giving the branded engine a key-free public link (#852)
- **docs**: fix stale "JxBrowser 8.15.0" reference in `CLAUDE.md` (#852)

## Dependencies

- **JxBrowser** 9.1.2 â†’ 9.3.0 (Chromium 150), with renewed subscription license key; `build-chromium-branding.yml` default and `chromium-branding/params.json` version stamp bumped to 9.3.0 (#852)
- Removed the explicit **guava** dependency (its only consumer was the deleted `WebAuthnOrigin` Public Suffix List check) (#852)
